# OpenClaw Telegram Footer

## Purpose

Maintain the local Telegram footer customization for OpenClaw replies. Since 2026-05-12, all three channels (Telegram, Discord, Feishu) share a single footer content source: `~/.openclaw/footer-shared.mjs`.

The patch does these things in the agent-runner runtime:

1. **Adds an ESM import** of `footer-shared.mjs` ‚Üí `generateFooterLine()`
2. **Replaces `formatTelegramFooterLine`** with a thin wrapper: `generateFooterLine({ ...params, style: "telegram" })`
3. **Replaces `formatDiscordFooterLine`** with a thin wrapper: `generateFooterLine({ ...params, style: "discord" })`
4. **Patches the footer callsite** so `Context` uses refreshed session-store `totalTokens / contextTokens`, not provider prompt usage (`input + cacheRead + cacheWrite`).
5. The old helper functions (token amount, date, duration, path, model) remain in the bundle as dead code.

**To change footer format across all channels:** edit `~/.openclaw/footer-shared.mjs` ‚Üí restart Gateway. No re-patching needed.

Telegram has no Feishu-style card `note` metadata, so this is a visible text footer appended to the final reply line.

## Shared module

All footer content is defined in a single file:

```
~/.openclaw/footer-shared.mjs
```

This file exports `generateFooterLine(params)` and is imported by the agent-runner runtime (Telegram + Discord) and the Feishu monitor. To change footer format for all channels, edit only this file, then restart Gateway.

## Workflow

1. If runtime script is missing or stale, copy bundled script:

```bash
mkdir -p ~/.openclaw/scripts
cp <skill-dir>/scripts/patch-telegram-footer.sh ~/.openclaw/scripts/patch-telegram-footer.sh
chmod +x ~/.openclaw/scripts/patch-telegram-footer.sh
```

2. Run `--check` before applying.
3. Run `--apply` only when markers indicate the patch is missing.
4. The script adds an ESM import and replaces two function bodies; backs up the target bundle; runs `node --check`.
5. Restart Gateway after `--apply`.
6. The shared module (`~/.openclaw/footer-shared.mjs`) must exist before `--apply`. If missing, restore from the skill bundle.

## Notes

- The patch adds an ESM import to the runtime and replaces `formatTelegramFooterLine` / `formatDiscordFooterLine` with thin wrappers.
- The old hand-rolled helpers (token amount, date, duration, path, model) remain as dead code in the bundle; they are harmless.
- Appending the visual separator (`‚îÄ‚îÄ‚îÄ‚îÄ`) is handled by `appendUsageLine` in the same runtime; this is not affected by the shared module.
- Discord shares the same runtime, so `patch-discord-footer.sh` delegates to `patch-telegram-footer.sh`.
- If OpenClaw updates and the import or wrappers are lost, run `--check`; if missing, run `--apply` again.

## Debugging notes

- A successful provider-usage fetch does **not** prove Telegram is showing the footer; it only proves the usage source works.
- A missing footer in Telegram can still happen even when `responseUsageLine` is generated, because the footer is appended in the final send payload path.
- When debugging, prefer live evidence in this order:
  1. provider-usage module returns non-empty windows
  2. runtime generates a non-empty `responseUsageLine`
  3. Telegram delivery path still contains the appended footer text
  4. real user-visible test message confirms it

### Temporary debug log cleanup strategy

- Temporary logs are allowed only for short-lived diagnosis of footer generation or footer delivery.
- Use a unique prefix such as `[footer-debug]` so added logs are easy to grep and remove.
- Prefer the narrowest possible probes:
  - one log where `responseUsageLine` is generated
  - one log near `appendUsageLine(...)`
  - one log in the final Telegram send path if delivery is still in doubt
- After the issue is confirmed fixed, remove all temporary debug logs before considering the patch complete.
- Cleanup checklist:
  1. grep for the debug prefix in the patched runtime bundle
  2. remove all temporary `console.warn` / `console.log` probes added for diagnosis
  3. run `node --check` again
  4. restart Gateway
  5. send one real Telegram test reply to confirm behavior without debug noise
- Do not leave high-frequency debug logs in a long-running Gateway; they add noise and can hide real footer warnings.

## Safety

- Do not edit secrets or OpenClaw config for this workflow.
- Do not guess if code patterns changed; stop and report the blocker.
- Prefer backup + `node --check` + Gateway status verification.
